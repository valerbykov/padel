# Спек 1 — Турнир: запись парами + обогащённое приглашение (D + E1 + E2)

Дата: 2026-07-19 · Статус: дизайн на ревью · Процесс: superpowers:brainstorming → writing-plans

## Проблема

1. **Пары формируются случайно.** Парные форматы (King of the Court сейчас, Round Robin в E) собирают пары через `shuffle` в `buildKotHLadderStart` — из-за этого при записи непонятно, кто с кем играет. Люди хотят записываться заранее сформированными парами.
2. **Приглашение бедное.** Страница `/t/CODE` (`TournamentJoin.jsx`) — утилитарный список. Хочется афишу и больше полей (описание, время окончания, контакт-«ответственный»), как в приглашениях конкурентов.

## Рамки

**В спеке:** D (запись парами), E1 (быстрые поля), E2 (афишная карточка) — потому что всё живёт на одной странице `/t/CODE`.
**Вне спека (отдельные спеки):** E3 (уровни игрока), E4 (взносы на старте + валюта), E (Round Robin), микс-пары (M+F), реальные платежи. Реальную оплату деньгами НЕ делаем.

## Модель данных

### `tournament_players` — пары
- Добавить `pair_no integer` (nullable). Два ряда с одинаковыми `(tournament_id, pair_no)` = пара. **Открытая пара** = `pair_no`, у которого пока ОДИН ряд (второго ряда просто нет — заглушку не вставляем, т.к. `name NOT NULL`).
- Максимум 2 ряда на `(tournament_id, pair_no)` — гарантируем в RPC (проверка count перед вставкой). Дополнительно — страховочный частичный индекс/триггер (детали в плане).
- `pair_no` осмыслен только для парных форматов; в solo-турнирах остаётся `null`.
- Существующая колонка `seat` — НЕ трогаем (функционально не используется); пары держим на `pair_no`, а не на `seat`.

### `tournaments` — поля E1
- `description text` (nullable) — свободный текст + эмодзи.
- `ends_at timestamptz` (nullable) — время окончания (начало уже есть — `starts_at`). Валидация `ends_at > starts_at` на клиенте.
- `contact_name text`, `contact_link text` (оба nullable) — «ответственный» (имя + @tg/ссылка, опционально).

Все миграции — на прод через Supabase MCP (проект `ofewhhcwswjxvlqsygxu`), тест-луп до деплоя UI.

## Пары (D) — потоки и логика

### Запись (турнир в статусе `open`, парный формат)
- **Создать пару**: игрок берёт новый `pair_no` (`max+1`), становится единственным участником (открытая пара). Может сразу добавить второго — игрока лиги или гостя → пара полная.
- **Встать напарником**: выбирает открытую пару (у которой 1 участник) → вставляется ряд с тем же `pair_no`.
- **Две ссылки**: общая `/t/CODE` (список открытых пар + «создать пару») и персональная `/t/CODE?pair=N` (сразу в пару N, без выбора).

### RPC `join_tournament` (правим 3-арг оверлоад — его зовёт клиент)
Добавить необязательный `p_pair_no integer default null`. Логика:
- Определяем `v_is_pair := v_t.format = any(array['king_of_hill','round_robin','beat_the_box'])`.
- Профиль вступающего — строго из `auth.uid()` (как сейчас); анонимы — гость по имени.
- Сохранить текущие ветки идемпотентности (`already`) и привязки гостя по ВЕРИФИЦ. контакту (email/phone/telegram; привязка по имени уже убрана 2026-07-19).
- **Парный формат:**
  - `p_pair_no IS NULL` → СОЗДАТЬ пару: назначить `next := coalesce(max(pair_no),0)+1`, вставить вступающего единственным участником. **RPC возвращает назначенный `pair_no`** (`jsonb {ok, pair_no}`), чтобы клиент мог предложить «добавить напарника» и собрать персональную ссылку `/t/CODE?pair=<pair_no>`.
  - `p_pair_no = N` → ВСТАТЬ в пару N: guard «в паре < 2 участников» и «вступающий ещё не в ней» → вставка с `pair_no = N`; иначе `pair_full` / `pair_not_found`. Атомарно (против гонки двух напарников на одно место) — по образцу `join_game_slot`.
- **Solo формат:** `p_pair_no` игнорируется, плоская вставка (как сейчас).
- Ёмкость (`target_size` = число игроков) проверяется как сейчас.

### Организаторский ростер и добавление (в приложении)
- `addTournamentPlayer(trnId, entry, pairNo?)` — добавить в конкретную пару (закрыть открытое место) или создать новую пару. Так организатор/создатель сразу сажает второго (игрока лиги или гостя).
- `removeTournamentPlayer`: удаление одного из пары оставляет второго открытой парой (`pair_no` сохраняется). Удаление единственного участника — освобождает `pair_no`.
- Ростер рисуется СГРУППИРОВАННО по парам.

### Экран формирования пар на старте
Отдельный шаг перед «Запустить»: организатор видит всех, сгруппированных по парам, и может: посадить одиночку в открытую пару, поменять напарников местами, удалить. **«Запустить» заблокирована, пока есть открытое место или непарный игрок** (нужно чётное число, все в полных парах). Кнопки «Перемешать»/рандома НЕТ.

### Жеребьёвка `buildKotHLadderStart`
Вместо `shuffle(playerIds)` + нарезка — **брать готовые пары из `pair_no`** и раскидывать пары по кортам (пары пар на корт). Вход — пары (сгруппированные по `pair_no`), не плоский список id. Защитно: если у кого-то нет `pair_no` / пара неполна — исключение (не должно случиться из-за старт-гейта). Легаси: уже стартованные турниры не трогаются (матчи сформированы); нестартованные легаси-KotH проходят через новый экран формирования пар.

### Итоговая таблица по парам
Сделать парную агрегацию ЯВНОЙ: строка таблицы = пара (сумма очков/against/wins двух игроков), сортировка по очкам/дельте. Чемпион-пара подсвечивается (проп `championIds` в `StandingsTable` уже есть из прошлой итерации — 👑 + золото). Убирает нынешний постфактум-пересчёт пары в KotH.

## E1 / E2 — поля приглашения и афиша

- `/t/CODE` (`TournamentJoin.jsx`) и шапка турнира в приложении показывают: название, лигу, `starts_at`–`ends_at` (диапазон), место, формат, ёмкость, **описание**, **контакт**.
- **Афишная карточка** по макету (https://claude.ai/code/artifact/c6961786-6300-49a6-9139-8a1eb07d1ffc): градиент, кубок, чипы. Палитра — существующая (`--bg:#0a1612 … --lime:#c8ff2d`). Тёмная тема (страница и так `color-scheme:dark`). «Организатор-бренд» → у нас **«Лига»**.
- **Секция записи парами** на `/t/CODE`: список открытых пар («X ищет напарника» + «Встать»), «создать свою пару», прогресс. Залогинен/гость — как сейчас.
- Чип взноса на афише показывает `fee_per_player`, ЕСЛИ задан. В этом спеке — как сегодня (валюта/тумблер/ввод-на-старте приходят в **Спек 2 (E4)**; здесь взнос не переусложняем, но и `₽` руками нигде НОВОГО не хардкодим — используем существующее форматирование).

## i18n
Новые ключи в `ru`/`en`/`es` (`src/lib/i18n.js`): описание (лейбл/плейсхолдер), время окончания, контакт, «создать пару», «встать напарником», «{name} ищет напарника», лейблы экрана формирования пар, парные лейблы таблицы. **Никакого хардкод-русского в JSX** (ESLint-гейт). Бренды конкурентов (Lunda и т.п.) в продуктовой копии не используем.

## Клиентский API (`src/lib/tournamentApi.js`)
- `createTournament` — принимает `description`, `ends_at`, `contact_name`, `contact_link`.
- `addTournamentPlayer(trnId, entry, pairNo?)`.
- `joinTournamentByCode(code, name, pairNo?)` — пробрасывает `p_pair_no`.
- Хелперы: создать пару (`p_pair_no=null` на парном формате) / встать в пару (`p_pair_no=N`).
- `getTournament` / `getTournamentByCode` — возвращают `pair_no` у игроков и `description`/`ends_at`/`contact_*` у турнира.
- Хелпер парной агрегации таблицы (в `americano.js` рядом с `detailedStandings` или новый модуль).

## Тест-луп (Playwright MCP, локальный preview)
Создать парный турнир → «создать пару» → оставить открытой → напарник по общей ссылке (выбор из списка) → напарник по персональной `?pair=N` → экран формирования пар (блокировка старта при открытом месте) → старт → лестница KotH из пар → итоговая таблица по парам, чемпион подсвечен. Плюс гейты `build` + `lint`.
